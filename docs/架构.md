# 脉冲游戏引擎 架构

## 一、编程语言

**核心设计总纲**：以「性能优先、分层适配、无缝协同、保障开发效率」为核心原则，构建三级语言分层架构，完美平衡极致性能、开发效率与热更新需求，实现三层语言共享同一套ECS框架、被同一个Job System驱动，彻底解决跨语言绑定成本、性能损耗、能力割裂的行业痛点。

### 1. 三级语言分层架构设计

基于代码变更频率、性能要求、生命周期、开发效率，明确划分三层语言的定位、适用场景与选型参考，贴合国内工业化开发的成熟实践：

- **第一层：引擎核心层**

定位：引擎底层核心模块开发，变更频率极低，要求极致的裸金属性能，无GC、无运行时开销

适用场景：平台抽象层、内存管理系统、Job System、ECS核心、渲染底层、序列化框架等引擎基础底座

- **第二层：性能逻辑层**

定位：游戏核心玩法、固定业务框架、高频执行逻辑开发，变更频率较低，要求高性能、可控的内存管理，支持全平台AOT编译

适用场景：核心战斗框架、大世界流式管理、角色控制器、渲染管线逻辑、ECS核心System等对性能敏感的业务逻辑

- **第三层：热更逻辑层**

定位：高频迭代的运营活动、剧情内容、临时玩法、BUG修复等逻辑开发，变更频率极高，要求极致的开发效率与全平台热更新能力，类似传统的脚本语言

适用场景：运营活动、剧情任务、UI交互逻辑、热更修复、配置化玩法等需要频繁更新的业务逻辑

### 2. 全链路零开销跨语言互操作

彻底解决传统跨语言开发的绑定繁琐、性能损耗大的痛点，实现三层语言的无缝互调：

- 无需手动编写P/Invoke、SWIG等绑定代码，基于统一类型系统自动生成跨语言调用胶水代码，编译期完成，零运行时开销

- 三层语言的函数、数据、事件可直接互相调用/访问，支持泛型、委托、lambda等高级特性的跨语言传递

- 热更层代码可直接继承、重写性能层、引擎层的虚接口，无需额外适配，实现业务逻辑的无缝扩展与热修复

### 3. 三层与ECS/Job System的原生深度融合

完全满足「三层同时操纵同一个ECS，被同一个Job System驱动」的核心诉求，实现三层能力无差异：

- 三层语言均可定义、访问、修改ECS的组件、Entity，共用同一套ECS内存数据，无数据拷贝、无同步开销

- 三层语言均可编写ECS System，System均可实现并行切片执行，自动交由统一的Job System调度，与原生代码享有同等的并行执行能力与性能

- 三层语言均可直接向Job System派发并行任务，支持任务依赖、优先级、延续等所有调度特性，无任何能力阉割

- 热更层新增的组件、System可无缝接入ECS的查询、调度管线，无需重启应用即可生效，完美支持热更新与MOD开发

### 4. 分层可控的GC与内存管理

针对三层语言的不同定位，提供差异化的内存管理策略，兼顾性能与开发效率：

- 引擎核心层：完全无GC，所有内存手动管理，基于分层内存分配器实现全生命周期管控，杜绝任何非预期的运行时开销

- 性能逻辑层：支持GC完全关闭、手动管控、增量分代GC三种模式，可精细化控制GC触发时机与分帧执行时长，严格限制GC暂停时间，杜绝主线程卡顿

- 热更逻辑层：提供独立的增量分代GC策略，支持分帧GC、后台GC，与性能逻辑层的GC完全隔离，互不影响

- 全引擎统一的内存调试、泄漏检测能力，三层语言的内存分配均可追踪、可观测

### 5. AOT/JIT混合编译架构

兼顾极致性能与热更新需求，适配全平台的编译限制：

- 引擎核心层、性能逻辑层：全平台全量AOT编译到原生机器指令，实现极致的执行性能，无任何解释执行开销

- 热更逻辑层：支持解释执行或编译为字节码执行，支持JIT，支持AOT编译为原生代码（方便支持web平台）

- 原生支持增量编译、跨平台交叉编译

### 6. 面向数据的语言特性全层级支持

保障三层语言均可编写高性能的面向数据代码，无特性阉割：

- 三层语言均原生支持值类型、只读结构体、ref struct、栈分配、固定大小缓冲区、指针操作等面向数据的核心特性

- 提供统一的面向数据的基础库，包括SIMD算法库、容器库、数学库，三层语言均可直接调用，性能一致

- 强制编码规范检查，避免在高频执行路径中使用面向对象的高开销特性，保障高性能代码的开发规范

### 7. 安全

性能逻辑层和热更逻辑层的错误不会导致整个游戏崩溃

### 8. 脚本与DSL

剧情脚本、技能脚本等特定领域用途的DSL

## 二、高性能

**核心设计总纲**：以「面向数据、极致并行、零冗余损耗」为核心原则，从内存底层到顶层帧架构全链路优化，最大化跨平台硬件算力利用率，兼顾高端设备的性能上限与低端机型的流畅下限，为全引擎模块提供统一的高性能执行底座。

### 1. 分层内存管理系统（高性能核心根基）

实现全引擎多维度、场景化的分层内存分配体系，从根源解决内存碎片、缓存命中率低、分配开销大的核心性能问题：

- 针对不同生命周期的内存场景，提供专属分配器：帧分配器（帧内临时数据，帧尾统一释放，零碎片）、栈分配器（短生命周期线性分配）、池分配器（高频创建销毁的固定大小对象）、通用分配器（长生命周期动态内存）

- 内置跨平台大页内存支持、强制缓存行对齐、冷热数据分离机制，最大化CPU缓存命中率

- 配套全链路内存追踪、泄漏/越界检测、碎片治理能力，所有内存分配可观测、可管控

### 2. DAG驱动的多线程任务调度系统（Job System）

原生适配全平台多核架构，实现无锁、高吞吐、低延迟的任务调度，彻底释放多核算力，是全引擎并行化的核心载体：

- 基于有向无环图（DAG）自动解析任务依赖关系，无依赖任务实现全核并行调度，杜绝线程空闲与资源争抢

- 支持任务优先级分级、抢占式调度、分帧执行、主线程饥饿保护，适配不同场景的调度需求

- 原生支持嵌套任务、任务延续、批量任务派发，与ECS、渲染、资源管理等引擎模块深度融合，实现全引擎无阻塞并行化

- 跨平台自适应调度：根据设备核心数、大小核架构、性能模式自动调整线程池规模与调度策略，兼顾性能与功耗

### 3. ECS架构

以「数据驱动、面向缓存、天然并行」为核心设计，是游戏逻辑的核心执行框架，原生支持并行化执行：

- 极致优化的内存布局：同类型组件连续内存存储，支持冷热数据分离、缓存行对齐，实现零拷贝连续内存访问，最大化缓存命中率

- 高性能查询系统：支持All/Any/None/Disabled等复杂组件组合查询，查询结果可直接传入Job System与SIMD指令，零运行时开销

- 原生支持System并行切片执行：System可基于Chunk/Entity粒度自动拆分并行任务，交由Job System调度执行，无锁并行处理，彻底释放多核性能

- 内置组件生命周期管理、标签组件、禁用组件、事件组件等实用能力，兼顾性能与开发效率

### 4. 跨平台统一SIMD抽象与执行框架

实现指令级的极致性能优化，抹平跨平台指令集差异，与ECS、Job System深度融合，实现全引擎自动向量化：

- 统一跨平台SIMD抽象层，自动适配x86(SSE/AVX)、ARM(NEON/SVE)、主机专属指令集，上层仅需编写一套向量化代码，编译时自动生成对应平台的最优机器指令

- 与ECS架构原生融合：System批量处理组件数据时，可自动触发SIMD向量化执行，无需手动编写汇编级代码

- 支持编译器辅助的自动向量化，提供向量化友好的容器与算法库，降低高性能代码的开发门槛

### 5. 逻辑与渲染完全解耦的并行帧架构

基于双缓冲的帧数据同步机制，实现游戏逻辑帧与渲染帧完全并行执行，互不阻塞，最大化CPU多核利用率：

- 逻辑帧与渲染帧各自独立的帧率控制，支持渲染帧率高于逻辑帧率，提升画面流畅度

- 基于只读帧数据的无锁同步，逻辑帧仅写入双缓冲的当前帧数据，渲染帧仅读取上一帧的完整数据，杜绝多线程数据竞争

- 与Job System、ECS深度融合，逻辑帧的并行执行与渲染帧的命令录制可完全并行，无串行瓶颈

- 在不支持的平台也可切换为串行渲染

## 四、模块化

**核心设计总纲**：以「最小核心、可插拔、全链路可扩展」为核心原则，实现引擎核心与上层能力完全解耦，原生支持团队协作、内容生态建设、游戏长线运营的全生命周期工程化需求，包管理、MOD、热更底层能力完全统一，避免重复建设。

### 1. 引擎核心最小化与模块完全解耦架构

确立模块化的核心根基，从架构层面杜绝强耦合：

- 引擎核心仅保留最基础的运行时底座：平台抽象层(PAL)、内存管理系统、Job System、ECS核心、统一类型系统、序列化框架，无任何业务相关的强依赖

- 渲染、物理、动画、音频、网络等所有上层能力，均以独立模块的形式实现，严格遵循接口契约开发，支持完全替换、按需裁剪、启用/禁用，无需修改引擎核心代码

- 所有模块均遵循「依赖倒置原则」，仅依赖引擎核心接口，模块间无直接强依赖，可独立开发、独立编译、独立测试、独立发布

### 2. 语义化版本包管理系统

模块化的基础载体，为所有模块、MOD、热更内容提供统一的包管理底座：

- 完整支持语义化版本规范，实现依赖自动解析、版本冲突自动解决、循环依赖检测，保障包依赖的稳定性

- 支持公有/私有包源管理、本地/远程包混合使用，适配团队内部私有模块、第三方开源模块、官方模块的统一管理

- 支持包粒度的权限控制、增量更新、按需加载，与热更、MOD系统共用底层的包解析、加载、校验机制

- 原生支持CI/CD流水线对接，实现包的自动构建、版本发布、分发全流程自动化

### 3. 安全可控的MOD生态支持系统

基于包管理底座构建，原生支持UGC内容生态，支持资源/数据字段覆盖，同时具有安全性与扩展性：

- 完整的包覆盖机制：支持MOD包对基础包、其他MOD包的资源、数据字段、预制体、ECS组件配置进行增量覆盖，基于优先级明确覆盖规则，无需侵入原有包体代码

- 严格的MOD沙箱安全机制：内置签名校验、权限管控体系，可精细化限制MOD的可访问API范围、内存配额、线程资源，避免恶意MOD篡改核心逻辑、破坏游戏稳定性，适配联网游戏的合规需求

- 完善的MOD生命周期管理：支持MOD的热加载/热卸载、启用/禁用、依赖管理、加载顺序控制，无需重启应用即可生效

- 与ECS、数据驱动系统深度融合：MOD不仅可覆盖资源与数据，还可新增自定义组件、System、数据配置、渲染Pass，完全不侵入原有游戏代码，最大化UGC创作自由度

### 4. 全链路统一热更新系统

基于包管理底座构建，与MOD系统共用底层能力，实现游戏全内容的热更新支持，兼顾稳定性与灵活性：

- 全链路热更覆盖：支持代码、资源、数据配置、ECS组件/System、渲染管线、着色器变体等所有游戏内容的热更新，无热更盲区

- 原子性热更与容错机制：支持增量热更、断点续传、完整性校验、签名校验，热更过程全程原子化，热更失败自动回滚到上一个稳定版本，杜绝应用变砖

- 精细化热更管控：支持包粒度、文件粒度的按需热更，支持预下载、后台静默更新、强制更新/非强制更新策略，适配不同的运营需求

- 与ECS、Job System原生融合：热更后的代码可直接注册到ECS调度器与Job System，与原生代码享有同等的执行能力与性能，无额外运行时开销

### 5. 低耦合模块间通信与契约系统

为模块化架构提供配套的通信机制，严格控制模块间的依赖边界：

- 统一的接口契约系统：所有模块必须基于抽象接口开发，明确输入输出契约，支持接口版本兼容，模块替换无需修改上层调用代码

- 多模式低耦合通信机制：提供同步接口调用、异步事件总线、跨模块消息队列等多种通信方式，模块间仅通过契约接口与事件通信，无直接代码依赖

- 内置模块生命周期管理、依赖注入能力，简化模块的初始化、销毁、依赖管理，降低模块化开发的门槛
