# 脉冲游戏引擎 架构

## 一、编程语言

**核心设计**：三级语言分层架构，共享ECS框架和Job System，平衡性能、开发效率与热更新。

### 1. 三级语言分层

- **第一层：引擎核心层**

定位：引擎底层核心模块开发，变更频率极低，要求极致的裸金属性能，无GC、无运行时开销

适用场景：平台抽象层、内存管理系统、Job System、ECS核心、渲染底层、序列化框架等引擎基础底座

- **第二层：性能逻辑层**

定位：变更频率较低，要求高性能的游戏框架、玩法部分，支持全平台AOT编译

适用场景：游戏核心玩法、核心战斗框架、大世界流式管理、角色控制器、渲染管线逻辑、ECS核心System等对性能敏感的业务逻辑

- **第三层：热更逻辑层**

定位：变更频率极高，要求极致的开发效率与全平台热更新能力，类似传统的脚本语言，支持解释执行/JIT/AOT编译

适用场景：高频迭代的运营活动、剧情任务、临时玩法、UI交互逻辑、热更修复、配置化玩法等需要频繁更新的业务逻辑

### 2. 跨语言无缝互操作

- 统一类型系统自动生成胶水代码，零运行时开销
- 函数、数据、事件直接互调，支持泛型、委托、lambda跨语言传递
- 热更层可直接继承重写性能层、引擎层虚接口

### 3. ECS/Job System融合

- 三层同时操纵同一个ECS，被同一个Job System驱动
- 三层均可定义、访问、修改ECS组件和Entity，共用内存无拷贝
- 三层均可编写System，并行切片执行，由统一Job System调度
- 三层均可派发并行任务，支持依赖、优先级、延续
- 热更层新增组件、System无缝接入，支持热更新和MOD

### 4. 内存管理

- 引擎核心层：完全无GC，手动管理，分层分配器
- 性能逻辑层：GC可关闭、手动管控或增量分代，精细化控制
- 热更逻辑层：独立增量分代GC，分帧/后台GC，与性能层隔离
- 统一内存调试、泄漏检测，三层均可追踪

### 5. 编译架构

- 引擎核心层、性能逻辑层：全平台AOT编译到原生指令
- 热更逻辑层：解释执行/字节码/JIT/AOT（web平台）
- 原生支持增量编译、跨平台交叉编译

### 6. 面向数据特性

- 三层均支持值类型、只读结构体、ref struct、栈分配、固定缓冲区、指针操作
- 统一面向数据基础库（SIMD、容器、数学库）
- 编码规范检查，避免高开销特性

### 7. 安全

性能逻辑层和热更逻辑层错误不导致整个游戏崩溃

### 8. 脚本与DSL

剧情脚本、技能脚本等特定领域DSL

## 二、高性能

**核心设计**：面向数据、极致并行、零冗余损耗，全链路优化。

### 1. 分层内存管理

- 帧分配器（帧内临时，帧尾释放）、栈分配器（短生命周期）、池分配器（高频固定大小）、通用分配器（长生命周期）
- 大页内存、缓存行对齐、冷热数据分离，提升缓存命中率
- 内存追踪、泄漏/越界检测、碎片治理

### 2. Job System

- 自动解析任务依赖，无依赖任务全核并行
- 支持优先级、抢占式、分帧执行、主线程饥饿保护
- 嵌套任务、任务延续、批量派发，与ECS、渲染、资源管理深度融合
- 跨平台自适应调度，根据核心数、大小核、性能模式调整

### 3. ECS架构

- 同类型组件连续内存存储，冷热分离、缓存行对齐，零拷贝访问
- 高性能查询（All/Any/None/Disabled）
- System基于Chunk/Entity粒度自动拆分并行任务，无锁并行
- 支持组件生命周期、标签组件、禁用组件、事件等特性

### 4. 跨平台SIMD

- 统一抽象层，自动适配x86(SSE/AVX)、ARM(NEON/SVE)、主机指令集
- 向量化友好的容器和算法库

### 5. 逻辑渲染并行解耦

- 基于双缓冲的帧数据同步，逻辑帧与渲染帧完全并行
- 各自独立帧率控制，渲染帧率可不同于逻辑帧率
- 只读帧数据无锁同步，逻辑写当前帧，渲染读上一帧
- 与Job System、ECS深度融合
- 不支持平台可切换为串行

## 四、模块化

**核心设计**：最小核心、可插拔、全链路可扩展，包管理、MOD、UGC、热更底层统一。

### 1. 最小核心与模块解耦

- 引擎核心仅保留PAL、内存管理、Job System、ECS核心、统一类型系统、序列化框架
- 渲染、物理、动画、音频、网络等以独立模块实现，可替换、裁剪、启用/禁用
- 依赖倒置原则，模块仅依赖核心接口，模块间无直接依赖，可独立开发编译测试发布

### 2. 语义化版本包管理

- 完整语义化版本支持，依赖自动解析、版本冲突解决、循环依赖检测
- 公有/私有包源、本地/远程包混合使用
- 包粒度权限控制、增量更新、按需加载，与热更、MOD共用底层机制
- CI/CD流水线对接，自动构建、版本发布、分发

### 3. MOD生态支持

- 包覆盖机制：MOD可增量覆盖资源、数据字段、预制体、ECS组件配置
- MOD沙箱安全：签名校验、权限管控，限制API访问、内存配额、线程资源
- MOD生命周期管理：热加载/热卸载、启用/禁用、依赖管理、加载顺序控制
- 与ECS、数据驱动融合：MOD可新增组件、System、数据配置、渲染Pass

### 4. 全链路热更新

- 全内容热更：代码、资源、数据配置、ECS组件/System、渲染管线、着色器变体
- 原子性热更与容错：增量热更、断点续传、完整性校验、签名校验、失败自动回滚
- 精细化管控：包粒度/文件粒度按需热更，预下载、后台静默更新、强制/非强制策略
- 与ECS、Job System原生融合，热更代码无额外开销

### 5. 模块间通信与契约

- 统一接口契约系统，明确输入输出，支持版本兼容
- 多模式通信：同步接口调用、异步事件总线、跨模块消息队列
- 模块生命周期管理、依赖注入

## 七、渲染系统

五层架构

1. 平台适配层：封装所有主流图形API（vulkan/dx11/dx12/metal/opengles/webgpu/webgl），只做图形API的基础封装，手感接近vulkan
2. 调度执行层：Render Graph、自动放置内存栅栏、跟踪内存布局、驱动API执行
3. 概念抽象层：抽象出Shader、材质、网格、命令缓冲（CommandBuffer）、Renderer List等概念
4. 管线框架层：可编程渲染管线、多线程渲染，类似Unity的SRP
5. 业务渲染层：项目实际渲染逻辑，位于性能逻辑层，引擎提供通用管线，项目可扩展替换，类似Unity的URP
